# Space Desk Evolution Hard Filter Results

## Filter Scope

Generation 1 for Space phone surface, accepted baseline `docs/design/mocks/space-surface-visual-refinement-v1.md`. Filters applied: CET product truth, visible physical hierarchy, current Law of One, favorite tag semantics, sleep zone semantics, Learning continuity, implementation authority, and phone proof containment.

## Rejected Candidates

- `space-06`: rejected because the progress command board makes counters and charts the focal object before the current box, turning Space into a statistics surface.
- `space-07`: rejected because the twin boxes make favorite and sleep physical containers, directly violating favorite-as-tag and sleep-under-original-box semantics.
- `space-08`: rejected because the globe hides the concrete library/group/box/card hierarchy behind decorative orbiting nodes and would force future implementation to invent spatial physics.

## Surviving Candidates

- `space-01`: passed hierarchy, Law of One, state semantics, continuity, and containment filters, but did not materially beat the accepted baseline.
- `space-02`: passed hierarchy after adding parent box rails around the drawer, with a risk that card inspect becomes too dominant.
- `space-03`: passed hierarchy and platform extensibility filters, with a risk of feeling like a file browser.
- `space-04`: passed all filters and improved the current box first-read path while preserving state semantics.
- `space-05`: passed hierarchy and sleep/wake semantics, with a risk that sleep becomes too emotionally central.

## Product Truth Violations

`space-06` violates Space-as-physical-hierarchy by leading with progress metrics. `space-07` violates favorite and sleep semantics by making them independent containers. `space-08` violates concrete knowledge ownership visibility because users cannot see a stable box and contained cards before decorative layout.

## Layout Or Proof Violations

`space-08` would require dense orbit labels that cannot be proven inside a `393 x 852` phone frame without overlap. `space-06` fits the viewport but fails proof relevance because the rendered regions would prove a dashboard, not Space. `space-07` fits visually but its proof would encode the wrong product model.

## Notes For Mutation

Mutate from `space-04` and borrow `space-03` shelf spine context plus `space-05` quiet sleep alcove. Do not borrow `space-02` if it hides the parent box, and do not borrow any progress-card chrome from `space-06`.
