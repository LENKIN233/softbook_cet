# Map Carousel

## Candidate ID

msr-07

## Provenance

- Tool or model: Codex design synthesis
- Prompt: Phone reset exploring a horizontal sequence of library regions and current card cards.
- Source context pack: `context-pack.md`
- Artifact: `candidate-proofs/mobile-reset-candidate-proof.html#msr-07`
- Screenshots: `candidate-proofs/mobile-reset-candidate-proof.html#msr-07`

## Product Truth Fit

Partly preserves Space but fails the Learning-first entry. Browsing the map becomes the first action.

## Focal Object

The horizontal map strip.

## First-Read Path

Map strip -> selected card -> navigation -> action.

## Interaction Silhouette

The silhouette reads as carousel browsing, not a current-card operation.

## Spatial Model

Library areas are visible, but group / box / card hierarchy is too compressed and may become arbitrary browsing.

## State Language

Favorite and sleep risk becoming tile badges rather than physical states.

## Motion Causality

Motion follows carousel browsing, which is not the primary learning cause.

## Platform Strategy

Phone clipping and discoverability would be difficult. Tablet could work, but phone is the release priority for this run.

## Implementation Mapping

Would require a custom carousel-first shell and weakens current Learning implementation mapping.

## Known Risks

It looks distinct but can become hard to operate and not mainstream enough for the requested app feel.

## Design Review Checklist Answers

Q1: Current library may be the active library, but neighboring library colors could easily violate Law of One.

Q2: Focal object is the map strip, not the active card.

Q3: Space silhouette is visible but Learning interaction silhouette is weak.

Q4: No forbidden visual pattern is necessary, but the interaction model fails.

Q5: Phone containment is risky because the carousel invites horizontal overflow.

Q6: Learning would drift toward browsing and module selection, so this candidate is rejected.
